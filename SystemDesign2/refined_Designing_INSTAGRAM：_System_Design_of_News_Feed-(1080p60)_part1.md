# Designing Instagram

When approaching a system design interview for a platform like Instagram, it is crucial to clarify the exact features the interviewer expects to be designed. While you may be familiar with the application, time constraints typically prevent deep dives into all functionalities. Therefore, always ask the interviewer to define the scope.

## Core Features for Instagram System Design

For this design exercise, the following key features are considered, as they represent fundamental and challenging aspects of Instagram's architecture:

1.  **Store and Retrieve Images:** The core capability to upload and access image content efficiently.
2.  **Like and Comment Facility:** Users can express reactions (likes) and provide textual feedback (comments) on images/posts.
3.  **Follow Functionality:** Users can establish connections by following other users' accounts.
4.  **News Feed Publication:** Users can view a personalized stream of content from accounts they follow.

## 1. Storing and Retrieving Images

This feature is a common requirement in many system designs involving media. Key considerations for image storage and retrieval include:

*   **Cost-Effective Storage:** Utilizing a distributed file system or object storage solution (e.g., AWS S3, Google Cloud Storage) to ensure economical storage for a large volume of images.
*   **Fast Content Delivery:** Implementing a Content Delivery Network (CDN) to cache images closer to users, thereby ensuring quick access and low latency during retrieval.

For a more detailed understanding of image storage and retrieval, refer to previous discussions on similar systems (e.g., Tinder design), which cover aspects like image resizing, metadata management, and secure uploads.

## 2. Liking and Commenting on Images/Posts

This section delves into the design considerations for user interactions with posts, specifically focusing on likes and comments.

### Defining "Posts"

*   In the context of Instagram, an "image" uploaded by a user is considered a "post."
*   The system needs to manage likes and comments associated with these posts.

### Commenting Hierarchy and Complexity

A critical design decision for comments is the level of recursion allowed:

*   **Question:** Can users comment on a comment (i.e., nested or recursive commenting)?
*   **Instagram's Actual Behavior:** Instagram currently allows one level of reply (a comment on a post, and then a reply directly to that comment). However, it does not support replies to replies, which limits the depth of conversation and system complexity.
*   **Interview Simplification:** For this system design interview, to simplify the scope and avoid the complexities of managing deep recursive comment trees, we will assume that users **cannot** comment on a comment. All comments are directly associated with a post only.

### Liking Functionality

*   **Liking Posts:** Users can express approval by "liking" posts (images).
*   **Liking Comments:** Despite the simplification for comment hierarchy, the system *will* support users liking individual comments.

### ER Diagram for Likes and Comments (Continued)

To handle likes on both posts and comments, a `Likes` table needs to be designed carefully.

![Screenshot at 00:03:36](notes_screenshots/refined_Designing_INSTAGRAM：_System_Design_of_News_Feed-(1080p60)_screenshots/frame_00-03-36.jpg)

#### Likes Table Design Considerations:

*   **`Post ID` vs. `Parent ID`:** Instead of solely relying on `Post ID`, a `Parent ID` is more flexible as it can refer to either a post or a comment.
*   **Identifying Parent Type:** To differentiate between a post and a comment when using `Parent ID`, a `Type` field is essential.
    *   **Poor Design Example:** Embedding the type directly into the ID (e.g., `comment_123`) is not ideal.
    *   **Recommended Design:** A separate `Type` column (e.g., `VARCHAR` or `ENUM`) that stores "COMMENT" or "POST". This allows the system to correctly query the respective `Posts` or `Comments` table based on the `Parent ID` and `Type`.

![Screenshot at 00:04:56](notes_screenshots/refined_Designing_INSTAGRAM：_System_Design_of_News_Feed-(1080p60)_screenshots/frame_00-04-56.jpg)

*   **Attributes for `Likes` Table:**
    *   `Like ID`: Primary key for the `Likes` table (autoincrementing integer).
    *   `Parent ID`: Foreign key referencing either `Post ID` in the `Posts` table or `Comment ID` in the `Comments` table.
    *   `Type`: Indicates whether the `Parent ID` refers to a "POST" or a "COMMENT".
    *   `User ID`: Foreign key referencing the user who performed the like.
    *   `Timestamp`: Records when the like occurred.
    *   `Active`: A boolean flag (or `deleted_at_timestamp`) indicating if the like is still active (not unliked).

#### Retrieving Like Counts

To find the number of likes for a particular post or comment:

*   **Basic Approach (Inefficient for News Feeds):**
    *   Perform a `SELECT COUNT(*)` query on the `Likes` table, filtering by `Parent ID` and `Type`.
    *   Example: `SELECT COUNT(*) FROM Likes WHERE ParentID = [some_post_id] AND Type = 'POST';`
    *   **Issue:** For a news feed displaying many posts, this approach would involve numerous individual `COUNT` queries, leading to slow performance (`RAM slow`).

*   **Denormalization Attempt (Bad Design):**
    *   Adding a `likes_count` column directly to the `Posts` or `Comments` table.
    *   Increment this column whenever a like is added for that parent.
    *   **Issue:** This is an aggregation that doesn't inherently belong to the `Posts` or `Comments` table. It violates normalization principles and can lead to data inconsistencies if not carefully managed (e.g., race conditions during updates).

*   **Recommended Approach: Activity Table or Separate Aggregation:**
    *   Instead of storing `likes_count` directly in `Posts` or `Comments`, maintain this information in a separate `Activity` table or a dedicated caching layer.
    *   The `Activity` table could store aggregate counts for various activities (likes, comments, shares) related to a `Parent ID`.
    *   **Attributes for `Activity` Table (Example):**
        *   `Activity ID`: Primary key.
        *   `Parent ID`: References the `Post ID` or `Comment ID`.
        *   `Type`: Indicates "POST" or "COMMENT".
        *   `Likes Count`: Stores the total number of likes for that parent.
        *   `Comments Count`: Stores the total number of comments for that parent.
        *   `Last Updated Timestamp`: To track when the counts were last refreshed.
    *   This approach keeps the core `Posts` and `Comments` tables clean while providing efficient access to aggregate data.

### ER Diagram for Likes and Comments (Continued)

Expanding on the `Likes` table design, it's crucial to include a unique identifier for each like action.

![Screenshot at 00:08:04](notes_screenshots/refined_Designing_INSTAGRAM：_System_Design_of_News_Feed-(1080p60)_screenshots/frame_00-08-04.jpg)

*   **`Like ID`:** A unique identifier (e.g., `BIGINT` primary key) for each entry in the `Likes` table. This `Like ID` uniquely identifies the action of a specific user liking a specific post or comment at a given time.

**Querying the Designed Tables:**

With the proposed ER diagram, various information can be efficiently retrieved:

*   **Number of likes on a post:** Query the `ActivityAggregates` table using `Parent ID` (post's ID) and `Type = 'POST'`. Alternatively, `SELECT COUNT(*) FROM Likes WHERE ParentID = [post_id] AND Type = 'POST' AND Active = TRUE;`
*   **All comments for a post:** Query the `Comments` table using `Post ID`.
*   **All users who liked a specific post:** Query the `Likes` table using `Parent ID` (post's ID) and `Type = 'POST'`.
*   **All posts by a specific user:** Query the `Posts` table using `User ID`.

This design ensures that information is available in a structured and retrievable manner. The `Posts` table already includes a `User ID` to link each post to its creator.

**Designing ER Diagrams:**

*   Creating an ER diagram is an iterative process, often requiring changes and refinements as you consider how data will be stored and retrieved.
*   Maintain flexibility in your design to accommodate future requirements while keeping current needs in mind.
*   If future needs are uncertain, prioritize simplicity and ease of understanding in your initial table structures.

## 3. Follow Functionality

The third core feature for Instagram is the ability for users to follow one another. This can be modeled with a single table.

![Screenshot at 00:10:09](notes_screenshots/refined_Designing_INSTAGRAM：_System_Design_of_News_Feed-(1080p60)_screenshots/frame_00-10-09.jpg)

**Key Questions to Answer:**

1.  **Who follows user X?** (i.e., X's followers)
2.  **Which users does user X follow?** (i.e., X's followees/following)

**`Follows` Table Design:**

A single table can efficiently answer both questions by storing pairs of user IDs and the timestamp of the follow action.

| Column Name       | Data Type  | Description                                        |
| :---------------- | :--------- | :------------------------------------------------- |
| `Follower User ID`| `BIGINT`   | ID of the user who initiated the follow            |
| `Followee User ID`| `BIGINT`   | ID of the user being followed                      |
| `Timestamp`       | `DATETIME` | Time when the follow relationship was established  |

*   **Example:** If User A follows User B, an entry would be: `Follower User ID = A`, `Followee User ID = B`.

**Answering the Questions using the `Follows` Table:**

*   **Who follows user X?**
    *   Query: `SELECT FollowerUserID FROM Follows WHERE FolloweeUserID = X;`
    *   This retrieves a list of all `Follower User ID`s where `X` is the `Followee User ID`.

*   **Which users does user X follow?**
    *   Query: `SELECT FolloweeUserID FROM Follows WHERE FollowerUserID = X;`
    *   This retrieves a list of all `Followee User ID`s where `X` is the `Follower User ID`.

## Interview Strategy: Direct Answers for Simple Questions

During a system design interview, if you know the direct and efficient answer to a simple question, it is generally beneficial to provide it quickly.

*   **Efficiency:** Saves time for both the interviewer and the interviewee.
*   **Preparation:** Demonstrates strong preparation and understanding of fundamental concepts.
*   **Progression:** Allows the interviewer to move on to more advanced or complex topics that they truly want to explore, enabling a deeper discussion.
*   **Focus on Advanced Concepts:** This approach (providing concise answers for basics to delve into more advanced system design concepts) is often preferred, as it allows for a more comprehensive and sophisticated discussion within the interview's time limits.

## System Architecture Overview

When designing Instagram, especially as a mobile application, the interaction between the client (mobile app) and the server-side components is crucial.

### Gateway (Reverse Proxy)

The system introduces a **Gateway** as a central entry point for all client requests. This Gateway functions as a **Reverse Proxy** and encapsulates several critical functionalities:

![Screenshot at 00:11:51](notes_screenshots/refined_Designing_INSTAGRAM：_System_Design_of_News_Feed-(1080p60)_screenshots/frame_00-11-51.jpg)

*   **Request Routing:** The Gateway decides which internal service should receive an incoming request.
*   **Security Mechanisms:** It handles authentication tokens and other security measures.
*   **Protocol Conversion:**
    *   It accepts well-known public protocols from clients (e.g., HTTP, WebSockets via XMPP for messaging).
    *   It then converts these external protocols into an internal, proprietary protocol understood by the backend services.
    *   This conversion adds a layer of security by obfuscating how internal servers are accessed, preventing direct external calls using the internal protocol.

![Screenshot at 00:12:01](notes_screenshots/refined_Designing_INSTAGRAM：_System_Design_of_News_Feed-(1080p60)_screenshots/frame_00-12-01.jpg)

### Request Flow for `getUserFeed`

Consider a typical request, such as `getUserFeed` (which retrieves a user's news feed) with a `UserID` parameter:

1.  The client (mobile app) sends an HTTP `getUserFeed` request to the Gateway.
2.  The Gateway, acting as a reverse proxy, receives this request.
3.  The Gateway needs to determine which specific `User Feed Service` instance should handle this request.

### Load Balancer and Service Scaling

To ensure resilience and scalability, services like the `User Feed Service` run on multiple servers. This introduces the challenge of routing client requests to the appropriate server.

![Screenshot at 00:12:41](notes_screenshots/refined_Designing_INSTAGRAM：_System_Design_of_News_Feed-(1080p60)_screenshots/frame_00-12-41.jpg)

*   **Load Balancer's Role:** A **Load Balancer** is responsible for distributing incoming network traffic across multiple servers.
*   **System State Management:** The Load Balancer maintains a dynamic "snapshot" of the entire system, including:
    *   All available services.
    *   Their physical locations (servers/boxes).
    *   How to connect to them.
*   **Coordination and Snapshot Distribution:** The Load Balancer continuously coordinates with Gateways and other services. It periodically sends this system snapshot (e.g., every 10 seconds) to the Gateways.
*   **Gateway-Side Routing:** When the Gateway receives a `getUserFeed` request, it consults its local, periodically updated snapshot from the Load Balancer to identify the correct `User Feed Service` instance to forward the request to.
*   **Consistent Hashing for Load Balancing:** Routing decisions are often made using algorithms like **Consistent Hashing**. By hashing the `UserID` (or another consistent key), the Gateway can deterministically identify the specific `User Feed Service` box responsible for that user's feed, ensuring requests for the same user consistently go to the same server while distributing load evenly.

*   **User Feed Service:**
    *   **Purpose:** Provides a curated list of posts (e.g., the top 20) that a user needs to see in their news feed.
    *   **Scaling:** Runs on multiple servers for high availability and performance.
    *   **Dependencies:**
        *   **Posts Service:** Responsible for storing and retrieving post-related information. It provides APIs to fetch posts by `UserID`, by date, or other criteria.

### Retrieving the User Feed: Initial Approach and Optimizations

The primary goal of the `User Feed Service` is to efficiently provide a personalized news feed for each user.

#### Initial Naive Approach (Pull Model)

When a `getUserFeed` request arrives at the `User Feed Service`, a straightforward but inefficient method would be:

1.  **Identify Followed Users:** The `User Feed Service` queries the `Follow Service` using the current `UserID` to get a list of all users this `UserID` follows (`getUsersFollowedBy(UserID)`). This returns a `Set<UserID>`.
2.  **Fetch Posts from Followed Users:** For each `UserID` in the set of followed users, the `User Feed Service` makes a separate request to the `Posts Service` to retrieve their posts (`getPostsByUser(UserID)`). This would typically return a `Set<Posts>`.
3.  **Aggregate and Return:** All retrieved posts are then combined, potentially sorted (e.g., by timestamp), and returned to the mobile application via the Gateway.

![Screenshot at 00:15:12](notes_screenshots/refined_Designing_INSTAGRAM：_System_Design_of_News_Feed-(1080p60)_screenshots/frame_00-15-12.jpg)

#### Inefficiencies of the Naive Approach

*   **N+1 Problem:** If a user follows `N` other users, this approach results in `N` separate requests to the `Posts Service` in addition to the initial request to the `Follow Service`.
*   **High Network Load:** Numerous service-to-service calls increase network traffic.
*   **Poor Scalability:** Re-computing the entire feed for every `getUserFeed` request is computationally expensive and does not scale well with a large number of users or followed accounts. It puts a significant load on the `Posts Service` database for every feed request.

#### Optimizations for the Naive Approach

To mitigate some of the issues, even with the pull model:

1.  **Batch API for Posts:**
    *   Instead of `getPostsByUser(UserID)`, the `Posts Service` can expose an API like `getPostsByUsers(Set<UserID>)`.
    *   This allows the `User Feed Service` to fetch posts from all followed users in a single batched request, reducing network calls from `N` to 1.
2.  **Limit Number of Posts:**
    *   The `getPostsByUsers` API can also include parameters to limit the number of posts returned per user (e.g., `getPostsByUsers(Set<UserID>, limit=X)`).
    *   This reduces the data transferred over the network and the load on the `Posts Service` and its underlying database.

Despite these optimizations, the fundamental problem of re-computation for every request persists, making the pull model unsuitable for a highly scalable news feed.

### The Scalable Solution: Pre-computing the User Feed (Push Model)

To achieve scalability, the user feed must be **pre-computed**. This means the feed is prepared in advance, so when a `getUserFeed` request comes in, the system can return it almost instantly.

#### How to Pre-compute the User Feed

The pre-computation process follows a **push model**:

1.  **New Post Creation:** When a user creates a new post (e.g., `Post(UserID, ImageURL, Text)`), this request goes to the `Posts Service`.
2.  **Post Persistence:** The `Posts Service` persists the new post data into its database.
3.  **Notification to User Feed Service:** After successfully persisting the post, the `Posts Service` sends a notification to the `User Feed Service` about the new post. This notification includes the `Post ID` and the `UserID` of the person who just posted.
4.  **Identify Followers:** The `User Feed Service` receives this notification. It then queries the `Follow Service` to find all users who follow the person who just posted (`getFollowersOf(PostCreatorUserID)`).
5.  **Incremental Feed Update:** For each follower identified, the `User Feed Service` adds the new post to their respective pre-computed user feed.
    *   This is an incremental update, significantly more efficient than recalculating entire feeds.
    *   Since user feeds are typically limited to a certain number of recent posts (e.g., the top 20), this "adding to a queue" operation remains efficient.

#### Storage for Pre-computed Feeds

A crucial decision is where to store these pre-computed user feeds:
*   **Database:** Storing them in a traditional database might introduce latency for reads if not optimized.
*   **Cache:** Using a fast in-memory cache (e.g., Redis) is generally preferred for pre-computed data that needs to be served quickly. The next step will detail this decision.

### Storage for Pre-computed User Feeds

The pre-computed user feeds need to be stored in a location that allows for extremely fast retrieval.

![Screenshot at 00:19:07](notes_screenshots/refined_Designing_INSTAGRAM：_System_Design_of_News_Feed-(1080p60)_screenshots/frame_00-19-07.jpg)

*   **Cache vs. Database:**
    *   **Cache (e.g., Redis):** Preferred for storing pre-computed user feeds.
        *   **Advantages:**
            *   **Speed:** Provides very low-latency access, crucial for a fast news feed experience.
            *   **Separation of Concerns:** Keeps the feed computation logic distinct from core data storage, reducing load on the primary database.
            *   **Recomputability:** If cache data is lost (e.g., service crash, cache eviction), the feed can be recomputed using the original brute-force method (querying `Follow Service` and `Posts Service`).
    *   **Database:** While possible, a traditional database might introduce higher latency for reads compared to an in-memory cache, especially for frequently accessed feeds.

*   **Cache Management for User Feeds:**
    *   **Finite Memory:** Caches have finite memory, so not all user feeds can be stored indefinitely.
    *   **Most Recently Used (MRU) Strategy:** Prioritize caching feeds for actively logging-in users.
    *   **Infrequent Users:** For users who log in infrequently:
        *   Their feeds might not be in the cache.
        *   When they do log in, their feed can be computed on demand (using the brute-force method).
        *   Once computed, their feed can be added to the cache, potentially displacing less recently used feeds.
    *   This strategy optimizes memory usage by only caching feeds for active users.

### Real-time Notifications for New Posts

The `User Feed Service` can also be leveraged to provide real-time notifications to users when new content is available from someone they follow.

*   **Notification Mechanism:**
    *   When the `User Feed Service` receives a new post and updates the pre-computed feeds of relevant followers, it can also send notifications to these followers.
    *   This requires a mechanism for the server to initiate communication with the client.
*   **Client-Server Communication Options:**
    *   **Polling:** Clients (mobile phones) periodically send requests to the server (e.g., every 10 seconds) to check for new content.
        *   **Disadvantages:** Can be inefficient, consumes more bandwidth and battery life on the client, and introduces latency.
    *   **Push Notifications (Server-Initiated):**
        *   **WebSockets:** A persistent, bidirectional communication channel between client and server, ideal for real-time updates. This is a favored method.
        *   **Long Polling:** The client makes a request, and the server holds the connection open until new data is available or a timeout occurs. The client then re-establishes the connection.
        *   Other proprietary push notification services (e.g., Apple Push Notification Service, Firebase Cloud Messaging) can also be used.
*   **Routing Notifications:**
    *   A challenge arises when a single post needs to be pushed to many followers (e.g., 50-60 users).
    *   This involves efficiently routing server-side notifications back to specific client devices. This complex topic, especially for a large number of recipients, is typically handled by specialized chat or notification services and will be discussed in more detail in future videos (or can be referenced in related videos like Tinder's design).

### The "Celebrity Problem": Handling Massive Fan-out

A significant challenge in the push model for news feeds arises with "celebrity" accounts that have millions of followers.

![Screenshot at 00:22:13](notes_screenshots/refined_Designing_INSTAGRAM：_System_Design_of_News_Feed-(1080p60)_screenshots/frame_00-22-13.jpg)

*   **The Problem:** If a celebrity posts, the `User Feed Service` attempting to push this new post to millions of followers simultaneously would create an enormous "fan-out" event. This sudden surge of updates and notifications can easily overload and crash the system.

*   **Solutions for the Celebrity Problem:**

    1.  **Batch Processing (Rate-Limited Push):**
        *   Instead of pushing to all followers at once, notifications/feed updates are sent in smaller batches.
        *   Example: Send updates to 1,000 followers every 10 seconds.
        *   This effectively rate-limits the outgoing requests, preventing system overload. However, it introduces a delay for some followers.

    2.  **Hybrid/Pull Model for Celebrities:**
        *   For celebrity accounts, switch from a push model to a pull model for their followers.
        *   **Mechanism:** When a celebrity posts, the system does *not* proactively update the feeds or send notifications to all their followers.
        *   **Client-Side Behavior:** Instead, the applications of the celebrity's followers (often highly engaged fans) periodically poll the server specifically for updates from that celebrity.
        *   **Advantages:**
            *   **Controlled Load:** The load on the server is distributed over time as users pull updates at their own pace, rather than being hit by a massive, simultaneous push.
            *   **Scalability:** Avoids the "fan-out" problem by shifting the responsibility of checking for updates to the client.
            *   **Resource Efficiency:** Prevents wasting resources on pushing updates to inactive users or those who might not check immediately.

**Comparison of Push vs. Pull Models for Celebrity Feeds:**

| Feature         | Push Model (Batch/Rate-Limited)                          | Pull Model (for Celebrity Followers)                     |
| :-------------- | :------------------------------------------------------- | :------------------------------------------------------- |
| **Update Delivery** | Server initiates updates to batches of followers.        | Client initiates requests to check for updates.          |
| **Latency**     | Varies (some followers receive updates sooner, others later due to batching). | Depends on client polling frequency.                     |
| **Server Load** | Managed by rate-limiting, but still involves server-side work for each batch. | Distributed over time, as clients pull at their own pace. |
| **Complexity**  | Requires batching logic and potentially a queuing system. | Simpler server-side logic; client handles polling.       |
| **Use Case**    | Suitable for large but manageable follower counts, or when immediate updates are critical for most users. | Ideal for extremely large follower counts where immediate, synchronous updates to everyone are infeasible. |
</REFINEDNOTES>

### The "Celebrity Problem": Handling Massive Fan-out (Continued)

#### The Push vs. Pull Debate

When considering the distribution of new posts to followers, there's a fundamental trade-off between **push** and **pull** models:

*   **Push Model (Fan-out on Write):**
    *   **Mechanism:** When a user posts, the system actively pushes that post to the feeds of all their followers.
    *   **Advantage:** Provides a seamless, real-time experience for followers, as updates appear instantly.
    *   **Disadvantage:** Prone to the "fan-out problem" for users with many followers (celebrities), leading to system overload if not carefully managed (e.g., with batching and rate-limiting).

*   **Pull Model (Fan-out on Read):**
    *   **Mechanism:** When a user wants to view their feed, their client actively pulls the latest posts from the server. The server then aggregates posts from the users they follow.
    *   **Advantage:** Distributes the load over time, as not all users will pull updates simultaneously. It avoids the massive, instantaneous fan-out associated with the push model.
    *   **Disadvantage:** Can lead to higher latency for updates (users only see new posts when they explicitly refresh) and potentially more frequent requests from active users, leading to higher read load on the server.

![Screenshot at 00:22:59](notes_screenshots/refined_Designing_INSTAGRAM：_System_Design_of_News_Feed-(1080p60)_screenshots/frame_00-22-59.jpg)

#### The Optimal Solution: A Hybrid Model

The most effective strategy for a platform like Instagram is to implement a **hybrid model**, combining the strengths of both push and pull:

*   **For Normal Users (fewer followers):** Employ the **push model**. This ensures a seamless, real-time experience for the majority of users, where new posts from their relatively small number of followees appear quickly.
*   **For Celebrities (millions of followers):** Employ the **pull model**. When a celebrity posts, the system does not attempt to push updates to all millions of followers immediately. Instead, followers' applications periodically pull for updates from these high-volume accounts. This prevents system crashes due to excessive fan-out and distributes the load more effectively.

### News Feed Architecture Summary

The architecture for the news feed service, incorporating pre-computation and a hybrid push/pull model, is a robust solution for scalable social media platforms.

*   **Pre-computation:** User feeds are built and updated incrementally in a cache (e.g., Redis) whenever a new post is created. This ensures fast retrieval.
*   **Follower List (Queue/List):** For each user, a list or queue of recent posts from their followees is maintained, typically limited to a specific size (e.g., 20 posts).
*   **Hybrid Delivery:**
    *   **Push:** For regular users, new posts are pushed to their pre-computed feeds, and notifications can be sent.
    *   **Pull:** For celebrities, followers' clients pull updates, reducing server load during massive fan-out events.

This architecture is widely adopted by popular applications requiring a dynamic news feed, such as Facebook, Twitter, Quora, and Instagram.
